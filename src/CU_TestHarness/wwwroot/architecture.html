Resource Architecture

DI uses a single AI Services account. CU uses two accounts — one hosts the CU endpoint, the other hosts GPT and embedding model deployments. CU calls the Models account internally via managed identity RBAC.

CU Analyzer Taxonomy

CU uses analyzers (not “models” like DI). They fall into three tiers based on whether an LLM or embedding model is needed. Click each tier to see examples.
📖 CU analyzer concepts · DI model overview

No LLM Tier 1 — Content-Only Analyzers

Pure OCR extraction. No model call needed — equivalent to DI's basic capabilities.

  • prebuilt-read — plain text extraction · DI Read model
  • prebuilt-layout — text + tables + structure · DI Layout model

These work identically whether or not the Models account has any deployments.

No LLM Tier 2 — Domain-Specific Analyzers

Pre-trained ML extraction for specific document types. No LLM needed — similar to DI prebuilt models.

  • prebuilt-invoice — invoices · DI Invoice model
  • prebuilt-receipt — receipts · DI Receipt model
  • prebuilt-document — general key-value pair extraction

Fixed output schema defined by Microsoft — you cannot add custom fields to these. Full list of DI prebuilt models →

GPT + Embeddings Tier 3 — LLM-Powered Analyzers

Require GPT and/or embedding model deployments in the Models account. This is what makes CU different from DI.

  • prebuilt-documentSearch — RAG-ready document search with embeddings
  • prebuilt-imageSearch — image-based semantic search
  • prebuilt-audioSearch — audio transcription + search
  • prebuilt-videoSearch — video understanding + search
  • Custom analyzers — user-defined schemas with extract, generate, or classify field methods · Create custom analyzers →

Prebuilt search analyzers resolve models via the PATCH /defaults endpoint (prebuilt-analyzer-completion, prebuilt-analyzer-embedding).
Custom analyzers specify models in their JSON "models" block.

Data Residency

Model deployment type determines whether inference data stays within Canada. This matters for regulated workloads.
📖 Azure OpenAI deployment types · Global Standard vs Standard

🍁 Standard Canada Guaranteed

Inference stays in the deployed region (Canada East). Use for data-sovereign workloads.

  • gpt-41-mini-ca — GPT-4.1 Mini Standard (Canada East)
  • gpt-4o-ca — GPT-4o Standard (Canada East)
  • text-embedding-ada-002 — Ada 002 (Canada East)
  • text-embedding-3-large — Embedding 3 Large (Canada East)
  • text-embedding-3-small — Embedding 3 Small (Canada East)

🌐 Global Standard May Leave Canada

Inference may be routed outside Canada for capacity. Higher throughput, lower cost per token.

  • gpt-41 — GPT-4.1 Global Standard
  • gpt-41-mini — GPT-4.1 Mini Global Standard

⚠ Prebuilt analyzer defaults currently point to Global Standard deployments (gpt-41, text-embedding-3-large). For full Canada residency, update defaults to use the -ca variants.
